# Security Audit Report - Deploily Backend
Date: 2026-04-07

## 1. Executive Summary
The security audit of the `deploily-backend` project has identified several critical and high-risk vulnerabilities that must be addressed before production deployment. The most significant issues include disabled CSRF protection, hardcoded default secrets, and unauthenticated access to infrastructure components (etcd).

## 2. Dependency Audit
The following outdated or potentially vulnerable dependencies were identified in `requirements.txt`:
- **Flask (2.2.5)**: Current version is 3.x. Older versions may have known vulnerabilities.
- **Werkzeug (3.0.6)**: Should be monitored for security advisories.
- **Pillow (9.1.1)**: Known vulnerabilities exist in older versions of Pillow (e.g., CVE-2023-44271). Recommended update to 10.x.
- **Requests (2.32.2)**: Generally safe, but always good to keep updated.

*Note: `pip-audit` failed to complete due to build issues with `pillow`, suggesting that the build environment for security scanning needs stabilization.*

## 3. Secret Scanning Results
The following potential secrets or weak defaults were found:
- **`src/config.py`**:
    - `SECRET_KEY`: Hardcoded default value `"abcdefghijklmnopqrtu"`.
    - `APISIX_API_KEY`: Hardcoded default value `"edd1c9f034335f136f87ad84b625c8f1"`.
- **`README.md` & `.devcontainer/.env.example`**:
    - `KEYCLOAK_CLIENT_SECRET`: Hardcoded test secret `"aYZTRwWfyLTRfqmBsqkfMJ9C68wSp0bO"`.

## 4. Infrastructure & Configuration Vulnerabilities
- **CSRF Protection Disabled**: `CSRF_ENABLED = False` in `src/config.py`. This exposes the application to Cross-Site Request Forgery attacks.
- **Unauthenticated etcd**: `.devcontainer/docker-compose.yml` sets `ALLOW_NONE_AUTHENTICATION: 'yes'` for the etcd service.
- **Internal IP Exposure**: `FAB_OPENAPI_SERVERS` in `src/config.py` lists multiple internal IP addresses (e.g., `192.168.1.x`), which can aid an attacker in mapping the internal network.
- **Swagger UI Exposed**: `FAB_API_SWAGGER_UI = True` is enabled by default. While useful for development, it should be disabled or restricted in production.
- **Insecure Public Key Fetching**: `fetch_keycloak_rs256_public_cert` in `src/config.py` uses `urllib.request.urlopen` without explicit SSL verification and lacks robust error handling if the identity provider is unreachable.

## 5. Logic & Code Review
- **`CustomSsoSecurityManager.load_user_jwt`**:
    - Automatically creates a `PaymentProfile` and adds `DEFAULT_CREDIT_AMOUNT` to new users. Ensure this is the intended behavior and cannot be abused by automated registrations.
    - Uses `email` as the primary identifier for user lookup. Ensure that the OIDC provider (Keycloak) guarantees email uniqueness and verification.

## 6. Recommendations & Hardening
1. **Enable CSRF Protection**: Set `CSRF_ENABLED = True` and ensure all frontend forms include CSRF tokens.
2. **Remove Hardcoded Secrets**: Use `os.environ.get("SECRET_KEY")` without a fallback, or use a strong, randomly generated fallback only in development.
3. **Secure Infrastructure**: Disable unauthenticated access to `etcd` and `redis`.
4. **Update Dependencies**: Upgrade `Flask`, `Pillow`, and other libraries to their latest stable versions.
5. **Sanitize Configuration**: Remove internal IP addresses from production configuration. Disable Swagger UI in production.
6. **Implement Proper Error Handling**: Ensure the application handles failures of external services (Keycloak, APISIX) gracefully without leaking sensitive information.
7. **Production Cleanup**: Ensure `.env` files, `.git`, and `tests` are excluded from production builds (verified in Dockerfile, but needs continuous monitoring).

## 7. Final Status
**STATUS: FAILED**
Immediate action is required to resolve the CRITICAL issues (CSRF, Hardcoded Secrets, Unauthenticated Infrastructure).
